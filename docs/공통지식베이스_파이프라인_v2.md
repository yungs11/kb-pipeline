## 1. 개요

### 1.1 목적
담보신탁 업무 중 공통 지식베이스 처리 절차를 기술한다. 하나의 업로드 문서를 업로드 → 파싱 → 문서 가드(규칙 검사) → 청킹 → 적재 → 메타저장 → 그래프재구성 의 단계로 흘려 단일 저장소에 적재·검색 가능 상태로 만드는 과정을 기술한다. 단계별 추적·격리·정합성을 보장하는 것이 목적이다.

### 1.2 적용 범위
- 현업이 지식베이스를 올리는 "나의 지식베이스"에 적용한다.
- 파서와 청커는 특화 지식베이스와 공통으로 사용한다.

### 1.3 기본 원칙
- 문서에 가장 적합한 파이프라인 전략을 선택하여 지식 베이스를 만든다.
- 지식베이스 생성 시, 지식베이스를 조회/수정할 그룹의 선택이 필요하다. (그룹에 할당됨)
- 해당 지식베이스의 그룹원이 아닌 경우, 사용할 수 없다. 할당이 필요한 경우, 관리자한테 요청한다.

## 2. 시스템 구성 요약

각 구성요소는 업무 역할로 정의한다.

| 구성요소 | 역할 |
|----------|------|
| 파이프라인 관문 | 업로드된 문서를 업로드→파싱→가드→청킹→적재→메타저장→그래프재구성 단계로 순서대로 흘려보내고, 각 건의 진행 상태를 추적·관리한다. |
| 문서 가드 | 적재 전 전역 검증 관문. 규칙 위반 시 적재를 진행하지 않고 현업에게 사유를 알린다. |
| 파서 | 업로드 문서를 시스템이 읽을 수 있는 형태로 변환한다. 문서 확장자에 따라 가장 적합한 파서가 선택된다. |
| 청커 | 변환된 문서를 검색에 적합한 단위(청크)로 분할한다. 문서 특성에 따라 여러 분할 방식을 비교해 가장 적합한 방식을 자동 선택한다. |
| 지식엔진·저장소 | 청크에서 핵심 정보(개체·관계)를 추출하고, 의미 검색용 색인과 관계 그래프를 만들어 단일 저장소에 적재한다. 이후 검색 질의도 이 엔진이 처리한다. |
| 검색 | 적재된 지식베이스를 대상으로 질문에 답한다. 의미가 가까운 내용과 관계로 이어진 내용을 함께 찾아 근거와 함께 제시한다. |

## 3. 전체 프로세스 흐름

### 3.1 단계 흐름 (한 줄)

```
업로드 → 파싱 → 문서 가드(규칙 검사) → 청킹 → 적재 → 메타저장 → 그래프재구성
```

위 단계와 별개로, 적재가 끝난 지식베이스에 대해서는 언제든 **검색(조회)** 이 가능하다

### 3.2 흐름 설명
- **업로드**: 현업이 처리할 문서를 "나의 지식베이스"에 올린다. 동일한 그룹원만 접근이 가능하다. 각 건에는 고유 식별값이 부여되어 이후 단계가 추적된다.
- **파싱**: 문서 확장자에 맞는 파서가 문서를 시스템이 읽을 수 있는 형태로 변환한다. 표·그림 등도 의미를 잃지 않도록 보존한다.
- **문서 가드(규칙 검사)**: 중복 여부와 문서 형식 규칙, 개인정보 검출 여부를 검사한다. 규칙을 통과한 건만 다음 단계로 진행하고, 위반 시 사유를 알리고 중단한다.
- **청킹**: 문서를 검색에 적합한 단위로 분할한다. 문서에 가장 적합한 분할 방식을 자동 선택한다.
- **적재**: 청크에서 개체·관계를 추출하고 의미 검색 색인과 관계 그래프를 만들어 단일 저장소에 저장한다.
- **메타저장**: 각 청크의 출처·페이지·표/그림 포함 여부 등 메타정보를 저장하여 화면 표시와 추적이 가능하게 한다.
- **그래프재구성**: 누적된 관계 그래프를 주제 단위로 재정리하고 주제별 요약을 생성하여, 전체를 아우르는 질의에 대비한다. (엑셀 문서는 제외)

### 3.3 처리 경로 / 반려·회귀 규칙
- **단계 추적**: 각 건은 업로드 시 부여된 고유 식별값으로 현재 어느 단계에 있는지(처리·대기·실패) 추적된다.
- **반려/회귀**: 문서 가드에서 규칙 위반으로 걸리면 적재를 진행하지 않고 현업에게 사유를 알린다(예: 엑셀 양식 오류). 파싱·적재 중 오류가 나면 해당 건을 실패로 표시하며, 재처리 여부는 운영 정책에 따른다.
- **로그**: 지식베이스 처리 혹은 삭제 이력을 모두 저장한다.
- **부분 실패 허용**: 표·그림의 의미 보강이나 페이지 이미지 생성처럼 본문 적재에 치명적이지 않은 처리는 실패하더라도 본문 적재는 계속 진행한다.

## 4. 단계별 상세 정의

### 4.1 업로드

| 항목 | 내용 |
|------|------|
| 수행 주체 | 현업(담당자) |
| 입력 | 처리할 원본 문서 |
| 처리 | 현업이 문서를 "나의 지식베이스"에 올린다. 업로드 시 각 건에 고유 식별값이 부여되어 이후 단계가 추적·관리된다. |
| 산출물 | 접수 건(고유 식별값 부여) |

### 4.2 파싱

| 항목 | 내용 |
|------|------|
| 수행 주체 | 시스템(파서) |
| 입력 | 업로드된 원본 문서 |
| 처리 | 문서 확장자에 맞는 파서를 자동 선택해 문서를 시스템이 읽을 수 있는 형태로 변환한다. 일반 PDF는 페이지 단위로 변환하고, 스캔본·이미지 문서는 글자 인식(OCR) 경로로 처리한다. 표는 구조(병합 셀 등)를 유지하고, 표·그림 등 비텍스트 요소도 검색에 활용할 수 있도록 의미를 보존한다. |
| 산출물 | 변환된 문서(페이지·구조 정보 보존) |

### 4.3 문서 가드 (규칙 검사)

| 항목 | 내용 |
|------|------|
| 수행 주체 | 시스템(문서 가드) |
| 입력 | 변환된 문서 |
| 처리 | 적재 전 전역 검증을 수행한다. 1. 이미 적재된 문서와 중복인지 확인하고, 2. 문서 형식이 처리 규칙에 맞는지 검사한다. 예를 들어 엑셀은 "머리행(헤더)–값" 형태로 추출 가능한지 검증하여, 규칙에 맞지 않으면 적재를 진행하지 않고 현업에게 보정 방법을 안내한다. 3. 개인정보 검출여부를 확인한다. |
| 산출물 | 검증 통과 건(다음 단계 진행) 또는 반려 안내 |

### 4.4 청킹

| 항목 | 내용 |
|------|------|
| 수행 주체 | 시스템(청커) |
| 입력 | 검증을 통과한 변환 문서 |
| 처리 | 문서를 검색에 적합한 단위(청크)로 분할한다. 여러 분할 방식을 비교·평가하여 해당 문서에 가장 적합한 방식을 자동 선택한다. 표·그림은 쪼개지지 않고 하나의 단위로 보존된다. |
| 산출물 | 청크 목록과 선택 근거(어떤 방식이 왜 선택됐는지) |

### 4.5 적재

| 항목 | 내용 |
|------|------|
| 수행 주체 | 시스템(지식엔진) |
| 입력 | 청크 목록, 대상 지식베이스 식별값 |
| 처리 | 각 청크에서 핵심 정보(개체·관계)를 추출하고, 의미 검색용 색인과 관계 그래프를 생성하여 단일 저장소에 저장한다. 모든 문서는 관계 저장까지 진행하되, 엑셀 문서는 그래프(관계)까지는 저장하지 않는다. |
| 산출물 | 적재 완료 건(검색 가능 상태) |

### 4.6 메타저장

| 항목 | 내용 |
|------|------|
| 수행 주체 | 시스템 |
| 입력 | 적재된 청크, 원본 문서의 페이지·구조 정보 |
| 처리 | 각 청크의 출처(원본 문서)·페이지 번호·표/그림 포함 여부 등 메타정보를 저장한다. 이를 통해 문서 상세 화면에서 청크별 위치·구성을 확인하고 추적할 수 있다. |
| 산출물 | 청크별 메타정보(화면 표시·추적용) |

### 4.7 그래프재구성

| 항목 | 내용 |
|------|------|
| 수행 주체 | 시스템 |
| 입력 | 누적된 관계 그래프(해당 지식베이스 범위) |
| 처리 | 적재로 쌓인 관계 그래프를 주제(커뮤니티) 단위로 재정리하고, 주제별 요약을 생성한다. 이는 개별 문장이 아니라 지식베이스 전체를 아우르는 질의에 답하기 위한 사전 정리다. 검색과 분리된 배경 작업으로 수행되며, 엑셀 문서는 대상이 아니다. |
| 산출물 | 주제별 묶음과 요약 |

### 4.8 검색 (조회)

적재 흐름과 별개로, 적재가 끝난 지식베이스에 대해 질의 시점에 동작한다.

| 항목 | 내용 |
|------|------|
| 수행 주체 | 시스템(검색) |
| 입력 | 대상 지식베이스 식별값, 질문 |
| 처리 | 질문과 의미가 가까운 내용과, 관계로 이어진 내용을 함께 찾아 근거와 함께 답을 구성한다. 검색은 항상 대상 지식베이스 범위로 한정되어, 다른 지식베이스의 내용이 섞이지 않는다(§5.4). |
| 산출물 | 답변과 근거(출처 청크 목록) |

## 5. 주요 프로세스 세부사항

각 단계가 내부에서 실제로 어떻게 동작하는지를 기술한다.

### 5.1 파싱 세부사항
- **확장자별 파서 자동 선택**: 문서 종류에 따라 처리 방식이 달라진다. 일반 PDF는 페이지·표 구조를 살리는 구조 파서로, PPTX와 이미지·스캔 문서는 글자 인식(OCR/AI 비전)으로, 엑셀은 엑셀 전용 파서로 보낸다.
- 현존하는 오픈소스 중에 확장자별로 가장 적합한 파서를 찾음. (이미지 추가)
-1. 확장자별 한국어 성능이 가장 뛰어나고 원문 보존이 정확한 파서를 사용한다.
2. 
- PDF: OpenDataLoader
- xlsx:
  - RAG에 필요한 엑셀 헤더:값에 대한 설명을 채워주는 RAG-Excel-Parser 사용(특화 파서 개발)
  - 전결규정의 경우, 하이라키 구조를 반영하여 같은 부모, 자식은 하나의 청크로 저장
    - kordoc:자산목록같은 표 구조 파싱(내부 반입시 hwp 모듈로 대체 가능)
    - openpyxl:전결규정 하이라키 구조 파싱
- image/scanned PDF:
  - 이미지 분석 필요: vlm (순서도 추론)
  - 텍스트 위주: ocr


- **페이지 이미지 보관**: 페이지 원문을 이미지로 떠서 함께 보관한다. 이후 문서 상세 화면에서 원문 페이지를 근거로 보여주는 데 쓰인다.

### 5.2 문서 가드 세부사항
- **중복 검사**: 이미 적재된 문서와 같은 내용인지 확인해 중복 적재를 막는다.
- **형식 규칙 검사**: 문서가 처리 규칙에 맞는지 검사한다. 특히 엑셀은 "머리행(헤더)–값" 형태로 추출 가능한지 확인하고, 맞지 않으면 적재를 멈추고 보정 방법을 안내한다.
- **개인정보 검출**: 문서에 개인정보가 포함됐는지 확인한다.
- 세 검사 중 하나라도 위반하면 다음 단계로 진행하지 않고 사유를 현업에게 알린다.

### 5.3 청킹 세부사항
- **여러 방식 경쟁 후 자동 선택**: 문서를 분할하는 여러 방식을 동시에 적용해 결과를 점수로 비교하고, 해당 문서에 가장 적합한 방식을 자동 선택한다. 어떤 방식이 왜 선택됐는지(선택 근거·점수)를 함께 남겨 화면에서 분할 품질을 확인할 수 있다.
- chunk 방식
    - 디폴트 제공 방식 설명
        1. **재귀 분할기(Recursive Splitter) 1100** "목표 토큰수(1100·600)에 도달하면 가장 가까운 구분자에서 자른다" 1100 토큰을 채우면 그 지점 근처 구분자(\n\n > 문장 > 단어 순)에서 끊는다. 
        2. **재귀 분할기(Recursive Splitter) 600** (target 600 토큰)은 같은 분할기를 더 작은 목표 크기로 돌려 더 세밀한 청크를 만든다. 
        3. **LLM 정규식 분할기 (LLM Regex Splitter)** 은 LLM에게 해당 문서(문서 앞 12장 정도)에 맞는 분할용 정규식 패턴을 생성하도록 요청한다. 
        4. **page**: 페이지 분할은 문서 페이지 경계를 기준으로 청킹한다.

│ BA     │ "제3조" 같은 조문 경계에서 깔끔하게 잘렸나 │
└────────┴────────────────────────────────────────────┘

- **화면에서 가장 적합한 청크 방식 선택 가능**: 사람이 방식 지정 가능. llm_regex 방식 선택 시, 사용자가 지정하는 정규식으로 청킹 가능
- **대용량·정교한 방식 대응**: 일부 정교한 분할 방식은 시간이 오래 걸리므로, 대용량 문서는 작업을 접수한 뒤 완료될 때까지 대기했다가 결과를 받는다.




### 5.4 적재 세부사항
- **핵심 정보 추출**: 각 청크에서 개체(인물·기관·용어 등)와 그들 사이의 관계를 인식해 관계 그래프의 재료로 삼는다.
- **의미 검색 색인(임베딩) 일관성**: 청크를 의미 기반으로 검색할 수 있도록 색인을 만든다. 분할·적재·검색 전 구간이 같은 방식의 색인을 사용해, 올릴 때와 찾을 때의 기준이 어긋나지 않게 한다.
- **엑셀 제외**: 모든 문서는 관계 저장까지 진행하되, 엑셀은 의미 검색 색인·메타정보까지만 저장하고 관계 그래프는 만들지 않는다.
- **진행 단계 추적**: 적재는 추출 → 색인 생성 → 저장 순으로 진행되며, 각 건이 현재 어느 단계인지 추적된다.

### 5.5 메타저장 세부사항
- **청크별 메타정보**: 각 청크에 출처(원본 문서)·페이지 번호·문서 내 순서·상위 제목 맥락·표/그림 포함 여부를 기록한다.
- **추적·연동**: 메타정보를 통해 문서 상세 화면에서 청크별 위치·구성을 확인하고, "그래프 보기" 등 관련 화면과 연동된다.

### 5.6 그래프재구성 세부사항
- **주제 단위 묶음**: 적재로 쌓인 관계 그래프를 서로 밀접한 개체끼리 주제(커뮤니티) 단위로 묶는다.
- **주제별 요약 생성**: 묶인 주제마다 한국어 요약을 만들어 둔다. 개별 문장이 아니라 지식베이스 전체를 아우르는 질의(요약·전체 동향 등)에 답하기 위한 사전 정리다.
- **기동 시점**: 검색과 분리된 배경 작업으로 수행되며, 수동 요청 또는 전체 요약형 검색 시 자동으로 만들어진다.

### 5.7 검색 세부사항
- **의미+관계 결합**: 질문과 의미가 가까운 내용(의미 검색)과 관계로 이어진 내용(관계 검색)을 함께 찾아 답을 구성하고, 출처 청크를 근거로 제시한다.
- **범위 한정**: 검색은 항상 대상 지식베이스 범위로만 수행되어 다른 지식베이스 내용이 섞이지 않는다.

### 5.8 격리·권한 (그룹)
- 지식베이스는 생성 시 지정한 그룹 단위로 격리되며, 조회/수정 권한은 그룹에 할당된다. 그룹원이 아니면 사용할 수 없고, 할당이 필요하면 관리자에게 요청한다.
- 적재·검색 등 모든 처리가 대상 지식베이스 범위로 한정된다.

## 6. 주요 협의사항

원활한 구축을 위해 다음 사항을 협의·확정한다.

- **권한 그룹 운영 정책** — 지식베이스–그룹 할당 방식, 조회/수정 권한 체계, 저장소 레벨 권한 강제 적용 시점을 확정한다.
- **반려·재처리 정책** — 문서 가드 반려 및 처리 실패 건의 알림 방식과 재처리 절차를 확정한다.
- **개인정보 검출 기준** — 문서 가드의 개인정보 검출 대상·기준과 검출 시 후속 처리(차단/마스킹/경고)를 확정한다.
