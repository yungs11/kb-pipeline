## 최우선 목적
**모든 형식의 엑셀 문서를 파싱 가능하게 하는 것.** 게이트는 "막는 도구"가 아니라 "추출이 깨지는 경우만 잡아 작성자에게 정확히 알려주는 도구".

핵심 가정(실측 근거): 현재 파서(기본 backend=kordoc)가 **헤더 후보군을 의외로 잘 찾아온다** (제목-위-헤더, 쌓인 표, 개념제목까지 잘 추출). 그러므로 **값이 잘못 뽑히면 = 헤더가 잘못 추출된 것**으로 간주하고, 사유와 위치를 알려 엑셀을 고치게 한다.

## 아키텍처 변경
1. **doc_guard(게이트웨이 검증)를 파서 뒤로 이동.** 파서가 추출한 내용을 가지고 검증한다(원시 바이트 아님).
   - UI 프로세스 순서도 `파싱 → 게이트웨이 검증 → 청킹 → 적재` 가 되게 한다.
2. **기존 doc_guard 13개 규칙은 전부 비활성.** 게이트웨이가 호출할 **신규 API 엔드포인트를 추가**하고, knowledge_base 는 그 신규 API 만 호출한다. (기존 `/v1/check` 경로는 더 이상 호출 안 함.)
3. **검증 계산 위치 = excel-parser.** excel-parser 가 시트별 **검증 요약**(추출오류/나란히2표/해당 행:열 위치 등)을 출력에 추가하고, doc_guard 신규 엔드포인트는 그 요약을 **받아서**(payload) 판정·메시지만 담당한다.

## 차단 규칙 (이 둘만 차단, 나머진 통과)
### 차단 ① 추출 실패 (값이 잘못 뽑힌 경우 = 헤더 오추출로 간주)
- **참조오류**: 셀 값에 `#REF! / #VALUE! / #DIV/0! / #N/A / #NAME? / #NULL! / #NUM!` 가 그대로 적재됨 (WBS 에서 확인)
- **헤더 누수**: 헤더행이 데이터 행으로 중복 적재됨 (헤더=값) (WBS 에서 확인)
- **빈 헤더 / 빈 컬럼명**: 헤더 컬럼명이 비어있거나, 값이 전부 비는 열이 생겨 header:값 매핑 실패
- → 위반 시 **사유 + 해당 행:열(셀 위치)** 을 표시하고 "이런 이유로 엑셀을 수정해주세요" 알림. 적재 멈춤(차단).

### 차단 ② 나란히 놓인 무관한 두 표
- 한 시트에 독립된 두 표가 좌우로 나란히 (예: 외부데이터 시트2 `순번|법령` + `순번|행정규칙`) → 거의 대부분 두 표를 한 행으로 잘못 병합 → **공식적으로 차단**.

### 통과 (차단하지 않음 — kordoc 이 잘 추출, 실측 확인)
- 제목-위-헤더 (헤더가 1행이 아니어도 OK): aws_cost_estimate(헤더 3행), 외부데이터 시트1(헤더 2행)
- 쌓인 표 / 표 안의 표 + 개념제목: 자산목록 "접근제어 적용 대상"
- 계층 구조 일반: WBS (단, #REF! 는 차단 ① 로 잡힘)
